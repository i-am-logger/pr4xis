# Dependability — Avizienis-Laprie-Randell-Landwehr (2004) taxonomy

Models the foundational taxonomy of dependable computing as a first-class ontology. Replaces the implicit `Result<(), Vec<String>>` pattern with a typed, citable understanding of what an Error IS — the precondition for the Resilience ontology (#123) that defines how systems recover from errors.

Per user direction surfacing PR #120 review:

> we need ontology understanding of erroring and recovery, backoff, etc

The central insight from Avizienis et al. (2004) is the **Fault → Error → Failure** chain: a Fault is the cause; an Error is the manifestation in system state; a Failure is what the user observes. Every dependable-systems concept hangs off this chain.

Key references:
- Avizienis, Laprie, Randell, Landwehr 2004: *Basic Concepts and Taxonomy of Dependable and Secure Computing*. IEEE TDSC 1(1), 11–33. — **the** foundational paper.
- Cristian 1991: *Understanding Fault-Tolerant Distributed Systems*. CACM 34(2). — Crash/Omission/Timing/Byzantine fault models.
- Lamport, Shostak & Pease 1982: *The Byzantine Generals Problem*. ACM TOPLAS 4(3). — origin of the Byzantine fault concept.

## Axioms

All six are first-class `Axiom` impls per `feedback_ontological_assertions.md` — discoverable via `Ontology::axioms()`, citable to the source paper:

| Axiom | Source | Claim |
|---|---|---|
| `ThreeThreats` | Avizienis §2.2 | Direct children of Threat are exactly {Fault, Error, Failure} |
| `FaultErrorFailureChain` | Avizienis §2.2 | Fault activates into Error; Error propagates into Failure |
| `FailureRecursionDocumented` | Avizienis §2.4 | Both Failure and Fault are Threats AND no Failure→Fault causal edge exists (recursion is inter-layer, not within-ontology) |
| `SixCoreAttributes` | Avizienis §4 | Direct children of Attribute are exactly the six core attributes |
| `FourMeans` | Avizienis §5 | Direct children of Means are exactly the four means |
| `CristianFaultModelsExist` | Cristian 1991 | Crash, Omission, Timing, Byzantine are all classified as OperationalFault |

## Functors

**Not yet.** A strict `Functor: Dependability → Diagnostics` was attempted and removed because the law `F(g ∘ f) = F(g) ∘ F(f)` fails: dependability is dense (44 concepts, no kinds) while diagnostics is kinded (10 concepts), and the many-to-one collapse breaks the law mechanically. The right structure is a lax functor or natural transformation, which the framework doesn't yet support. See `mod.rs` for the deferred note. Tracked as follow-up.

Direct relationships across ontologies that would benefit from such a functor:
- `Dependability → Diagnostics` (Reiter 1987): Failure → Symptom, Fault → FaultMode, ErrorDetection → Test
- `Dependability → Control` (Wiener): FaultTolerance is feedback control on Errors
- `Dependability → Observation` (JDL): failure detection through Observation morphisms
